# Production Data Audits — How-To

> Phase B2. The audit script
> [`backend/scripts/audit_production.py`](../../backend/scripts/audit_production.py)
> is the single entry point. It produces a markdown report with 12
> sections covering every collection that affects the v1 product. It
> is **read-only**: a code-level wrapper around the Motor database
> handle refuses every mutation method (insert / update / delete /
> drop / index ops). Even if the operator forgets to use a read-only
> Atlas connection string, the script still won't write.

## Prerequisites

1. **A read-only Atlas connection string.** Recommended over
   read-write — defense in depth.
   - Go to Atlas → Database Access → Add New Database User.
   - Authentication: SCRAM (password).
   - Built-in role: **`readAnyDatabase`** (or `read` scoped to the
     production database).
   - Generate a strong password; store in your password manager.
   - Network Access: ensure your laptop's IP is allowlisted.
   - Connect → Drivers → copy the `mongodb+srv://...` URI; replace
     `<password>` with your actual password.

2. **Python 3.11+** with the project's `requirements.txt` installed.
   The script uses `motor` (already a dep).

3. **Repo checked out** at the commit you want to audit against. The
   script reads `frontend/src/utils/notificationPresets.js` to keep
   its preset detection in lockstep with the JS source.

## Running the audit

From the repo root:

```bash
MONGO_URL='mongodb+srv://readonly:PASSWORD@cluster.mongodb.net/' \
DB_NAME='blueview' \
python -m backend.scripts.audit_production \
  > docs/audits/production-data-audit-$(date +%Y-%m-%d).md
```

The script:

- Validates env vars; exits 2 with a clear error if missing.
- Connects to Atlas via Motor.
- Runs **Section 7 first** (the filing_reps credentials canary). If
  any company doc still carries the legacy `filing_reps.credentials`
  field, the script aborts with **exit code 1** and a loud warning;
  the audit cannot complete until the operator runs
  `migrate_clear_filing_rep_credentials`.
- Runs the remaining 9 sections sequentially (~10–30 seconds total
  on a healthy cluster). Slow queries (>5s) log a warning to stderr.
- Buffers all output, then prints the final markdown:
  - **Executive Summary** at the top — count of blockers / warnings
    / info concerns + each concern formatted as a bullet.
  - **Section 11 (Atlas resource state)** — manual checklist the
    operator fills in from the Atlas dashboard.
  - **Sections 1–10** with detailed counts, distributions, and
    aggregation pipelines documented inline as code blocks.

## Section 7 canary failure

If you see this in the output:

```
BLOCKER: Section 7 canary failed — filing_reps.credentials field present.

filing_reps.credentials field present on N company doc(s). The MR.14 4b
cleanup migration must run before the audit can complete. Run:
`python -m backend.scripts.migrate_clear_filing_rep_credentials --execute`
```

Then:

1. Run the cleanup migration:
   ```bash
   MONGO_URL='mongodb+srv://...' DB_NAME='blueview' \
     python -m backend.scripts.migrate_clear_filing_rep_credentials --dry-run
   # Inspect the dry-run output — confirm the doc count.
   MONGO_URL='mongodb+srv://...' DB_NAME='blueview' \
     python -m backend.scripts.migrate_clear_filing_rep_credentials --execute
   ```
2. Re-run the audit. It should now reach Section 7 and emit
   "✅ CANARY CLEAN".

The cleanup migration is idempotent — safe to re-run even after a
clean execution.

## Read-only enforcement

Two layers:

1. **Operator-supplied creds** (recommended): use a connection string
   for a Mongo user with the `readAnyDatabase` role only.
2. **Code-level guard** (always-on): `ReadOnlyDatabase` wraps every
   collection access in `ReadOnlyCollection`, which raises
   `RuntimeError` on any banned method. The banned set is locked by
   `test_audit_production.py::TestBannedMethodsSet::test_membership_is_locked`
   so a future commit can't silently weaken the guard.

If the script accidentally calls a write method (e.g. due to a future
refactor adding an unintended `db.foo.insert_one(...)`), the
`RuntimeError` aborts the audit with a stack trace pointing at the
call site. No silent writes possible.

## How often to run

**Recommended cadence:**

- **Monthly**: routine health check. Catches drift between expected
  and actual production state — orphan users, stale digest queue
  records, missing indexes.
- **Before any major release**: especially anything touching
  schema, migrations, or tenant scoping.
- **After incident postmortems**: when the postmortem mentions data
  shape, confirm the audit reflects the corrective actions.
- **Before customer onboarding milestones**: greenlight only if the
  blocker count is 0.

Each audit run produces a new markdown file dated for the day of the
run. Commit the audit reports to `docs/audits/` so the history is
visible — they accumulate as a lightweight trend log.

## Re-running individual queries

The audit's aggregation pipelines are documented inline in the
markdown report. To re-run any single query without re-running the
whole audit, copy the pipeline into `mongosh`:

```bash
mongosh "$MONGO_URL/$DB_NAME" --quiet --eval '
db.projects.aggregate([
  { $match: { is_deleted: { $ne: true } } },
  { $group: { _id: "$company_id", count: { $sum: 1 } } },
  { $sort: { count: -1 } },
  { $limit: 10 }
]).toArray()
'
```

## Reviewing the report

After the script writes the markdown:

1. **Read the executive summary first.** If `BLOCKER` count > 0,
   stop. Address blockers before proceeding to the rest of the audit.
2. **Walk warnings in order.** Each carries a `Fix path:` —
   evaluate per-warning whether to address now or defer.
3. **Skim infos.** These are notable but rarely actionable.
4. **Fill in Section 11 (Atlas)** from the Atlas dashboard. Add
   any concerns to the executive summary manually.
5. **Decide which concerns block the next milestone.** For B3
   customer onboarding: blockers always block; warnings may block
   depending on severity.

Commit the audit report to `docs/audits/`. The filename pattern is
`production-data-audit-YYYY-MM-DD.md`. If you need to run multiple
audits in a day (rare), append a suffix:
`production-data-audit-YYYY-MM-DD-postfix.md`.

## When the audit script itself needs updating

The audit script is part of the codebase. If a new collection lands
or a new shape needs auditing:

1. Add a new `section_*` function in `backend/scripts/audit_production.py`.
2. Append it to `SECTION_RUNNERS`.
3. Update the test count assertion in
   `test_audit_production.py::TestScriptStaticPins::test_script_registers_all_section_runners`.
4. Re-run `pytest backend/tests/test_audit_production.py` to verify.
5. Commit the change. The next monthly audit picks up the new section.
